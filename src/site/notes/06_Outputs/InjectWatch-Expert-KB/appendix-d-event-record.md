## 为什么要立这个规范

现在的关井原因台账有四个治不了的老毛病：很多真实问题没记录（管线堵、漏的实际次数远多于台账）；事后补记，日期对不上；一挂一两年不更新；"其它停产井"成了什么都往里装的垃圾桶（1.5 万条，占全部记录近一半）。这些毛病的根源不是工人不认真，而是**没有规范**——没规定什么时候记、记什么、原因只能从哪些里面选。

本规范从源头解决。今后每次"系统提示 → 现场核查"，核查结果按本规范记录一条**事件记录**。这些记录有三个用处：核查过的问题有据可查，不再重复派工；积累"什么数据表现对应什么实际原因"的对照，系统据此逐步学会自动提示原因；判断标准（附表 A）修订时有实证依据。

**推行路径分三步，不冒进：** 第一步，本项目的核查工单先用（5 号干线试点，完全可控）；第二步，运行几个月后，拿"规范记录"与老台账对比展示效果；第三步，效果被认可后，由采油厂决定是否推广到日报制度。

## 记录单位：一个事件一条记录

**事件 = 一个对象 + 一段时间 + 一类问题。** 对象是一口井、一个配水间或一条干线；时间是从数据开始偏离到处理恢复（或确认无需处理）的整段。

三条铁律：

1. **不按天记。** 一个持续 20 天的欠注是一个事件，不是 20 条记录。
2. **不重复开。** 同一对象同一类问题，前一条事件没关闭，不开新的。
3. **不空着原因结案。** 每个事件关闭时必须有"确认原因"或明确写"未查明"，不允许空白。

## 分工原则：系统包办一切能从数据推出来的，人只补数据推不出的

工人和工程师的时间不该花在填表上。凡是能从日报数据和规则计算得出的内容，**一律由系统自动生成、自动更新**；现场人员只提供一样东西——**到现场看到的实际情况**，而且以点选为主，不写字。

## 字段定义（按来源分三类）

**第一类：系统自动生成，无需任何人动手（8 项）**

| # | 字段 | 怎么来 |
|---|------|--------|
| 1 | 事件编号 | 自动顺序号 |
| 2 | 对象 | 报警规则命中的井 / 配水间 / 干线 |
| 3 | 发现方式 | 命中的规则号（人工发现时才需手选） |
| 4 | 数据表现 | 命中的异常类型（附表 A 的 16 类），多规则同时命中自动并列 |
| 5 | 异常开始日期 | 回溯曲线上偏离起点，系统自动定，事后自动修正 |
| 6 | 发现日期 | 报警日期 |
| 11 | 恢复日期 | 系统持续盯着数据，恢复正常自动记日期 |
| 12 | 处理效果（初判） | 系统对照处理前后曲线自动给"恢复 / 部分恢复 / 未恢复"，人只需确认或改判 |

**第二类：系统预填，现场点一下确认（2 项）**

| # | 字段 | 怎么来 |
|---|------|--------|
| 7 | 核查日期 / 核查人 | 手机端打开事件即自动记时间，核查人即登录人 |
| 9 | **确认原因** | 系统按附表 C 把候选原因**排好序推给核查人**（如欠注 + 油压升 → 首选"近井堵塞"），核查人**点选**确认或改选其他项。选"其他"或"未查明"才需要写一句话 |

**第三类：只能现场提供（2 项，点选为主）**

| # | 字段 | 怎么填 |
|---|------|--------|
| 10 | 处理措施 | **系统按确认原因推荐排序**（如原因选"近井堵塞"→首推酸化/洗井；选"计量偏差"→首推校表），核查人点选确认或改选。全表可选：清管 / 校表换表 / 放套压 / 修井 / 调配注 / 修阀门水嘴 / 补管线 / 移交地质 / 无需处理 / 待批 |
| 8 | 现场所见（可选） | 拍照，或一句话（支持语音转文字）；不强制 |

**一次完整记录，现场人员的操作量：点开推送 → 点选原因 → 点选措施 → （可选）拍张照 → 提交。目标一分钟以内。**

## 客户端设想（集成进系统后）

系统报警后向核查人手机推送**事件卡片**：卡片上已带趋势曲线（标出异常时段）、命中的规则、按可能性排序的候选原因、建议核查步骤（附表 C 内容直接嵌入）。核查人到现场看完，在卡片上点两三下就完成记录。事件关闭由系统根据数据恢复自动发起，人只做最后确认。

系统上线前的过渡期，用同样字段的 Excel 模板过渡：系统生成部分由研发批量预填好发出，现场只填第二、三类。

## 原因词表（受控，按流程位置分组）

确认原因只能从下表选。词表与附表 C 的候选原因一一对应，保留了现场习惯用词。

| 流程位置 | 原因 |
|---------|------|
| 干线 | 干线结垢堵塞 · 干线渗漏 · 干线阀门卡滞 · 泵/上游供压不足 |
| 配水间 / 支线 | 支线堵 · 支线破漏 · 分水器偏流 · 改线未更新台账 |
| 井口 | 井口阀门/水嘴问题 · 井口漏 · 仪表失灵 · 抄录错误 |
| 井筒 | 近井堵塞 · 封隔器失封（串通）· 单流阀未复位（洗井憋压）· 窜槽 |
| 地层 | 地层吸水变差 / 不吸水 |
| 计划与外部 | 配注计划超出输水能力 · 计划作业（注明哪种）· 停电等外部因素 |
| 兜底 | 其他（必须写明）· 未查明（说明查了什么没查出来） |

**词表的修订：** 词表不是一次定死的。新原因出现，先记"其他"并写明，季度评审时决定是否入表。入表要同时补进附表 C 的核查指引。

## 质量要求

1. **"未查明"不是丢人的选项。** 如实点"未查明"比硬选一个原因有价值得多——错误的原因会教坏系统。
2. **数据表现与确认原因分开。** 第 4 项是数字上看到的（欠注，系统自动填），第 9 项是现场查实的（支线堵，人点选）——两者的对照正是这套记录最值钱的部分，也是系统今后学会自动判断原因的教材。
3. **"其他"占比超过 10% 触发词表评审。** 防止"其他"变成新的垃圾桶。
4. **系统性事件不拆单。** 干线问题开一条干线事件，受影响的井由系统自动关联列出，不给每口井单开。
5. **推荐要接受检验。** 系统推的**首选原因**与现场实际点选的一致率、**首推措施**与实际采用措施的一致率，均按月统计——这两个数字就是系统"智能归因、派工建议"水平的直接度量，也是推荐排序改进的依据。推荐永远只是排序，选择权在现场：工人可以随时展开全表按分类自选。

## 与现行台账的关系

现行日报、关井原因台账**照旧填报，本规范不替代、不冲突**。两者关系：台账是生产管理记录，本规范是异常核查记录。若将来采油厂决定统一，本规范可作为台账改版的底稿。

## 需要现场确认的问题

1. 原因词表是否有遗漏或不符合现场叫法的条目？
2. "核查人在手机端点选原因和措施"这个方式，现场是否接受？过渡期用预填好的 Excel 是否可行？
3. "其它停产井"类的历史遗留井，重新核查后是否按本规范补开事件记录？
